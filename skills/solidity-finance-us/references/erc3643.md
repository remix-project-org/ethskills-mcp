# ERC-3643 — T-REX (Token for Regulated EXchanges) Reference

## Overview
ERC-3643 is the most widely adopted standard for compliant security tokens in production,
developed by Tokeny Solutions. It extends ERC-20 with an on-chain identity layer and
modular compliance engine. Popular in EU (under MiCA/MiFID II) but used globally.

Key advantage over ERC-1400: simpler architecture, active open-source ecosystem,
and battle-tested with billions of dollars in tokenized assets.

GitHub: https://github.com/TokenySolutions/T-REX

---

## Architecture Overview

```
Token (ERC-3643)
│
├── Identity Registry          ← Who is allowed to hold tokens
│   ├── Identity Registry Storage
│   └── Trusted Issuers Registry  ← Which claim issuers are trusted
│
├── Compliance Module          ← What rules apply to transfers
│   ├── MaxBalance Module
│   ├── CountryRestrict Module
│   ├── MaxInvestors Module
│   └── (custom modules)
│
└── Claim Topics Registry      ← What claims are required to hold tokens
```

## Core Interfaces

```solidity
// IERC3643 — the token itself
interface IERC3643 is IERC20 {
    // Identity & compliance
    function identityRegistry() external view returns (IIdentityRegistry);
    function compliance() external view returns (IModularCompliance);

    // Forced transfers (for legal/regulatory compliance — e.g. court orders)
    function forcedTransfer(address from, address to, uint256 amount) external returns (bool);

    // Freeze / unfreeze (regulatory hold)
    function freezePartialTokens(address account, uint256 amount) external;
    function unfreezePartialTokens(address account, uint256 amount) external;
    function setAddressFrozen(address account, bool freeze) external;

    // Mint / burn
    function mint(address to, uint256 amount) external;
    function burn(address account, uint256 amount) external;

    // Recovery (lost wallet)
    function recoveryAddress(
        address lostWallet,
        address newWallet,
        address investorOnchainID
    ) external returns (bool);

    // Batch operations (gas optimization)
    function batchTransfer(address[] calldata toList, uint256[] calldata amounts) external;
    function batchForcedTransfer(
        address[] calldata fromList,
        address[] calldata toList,
        uint256[] calldata amounts
    ) external;
    function batchMint(address[] calldata toList, uint256[] calldata amounts) external;
    function batchBurn(address[] calldata accounts, uint256[] calldata amounts) external;
    function batchSetAddressFrozen(address[] calldata accounts, bool[] calldata freeze) external;
    function batchFreezePartialTokens(address[] calldata accounts, uint256[] calldata amounts) external;
}
```

## Identity Registry

```solidity
// Maps wallet address → ONCHAINID identity contract
// ONCHAINID: ERC-734/735 identity standard — stores claims about the identity

interface IIdentityRegistry {
    // Register identity: wallet → ONCHAINID address
    function registerIdentity(
        address investorAddress,
        IIdentity identity,
        uint16 country           // ISO 3166-1 numeric country code
    ) external;

    function deleteIdentity(address investorAddress) external;
    function updateCountry(address investorAddress, uint16 country) external;
    function updateIdentity(address investorAddress, IIdentity identity) external;

    // Check: does this wallet have a valid registered identity with required claims?
    function isVerified(address userAddress) external view returns (bool);
    function identity(address userAddress) external view returns (IIdentity);
    function investorCountry(address userAddress) external view returns (uint16);

    // Batch registration
    function batchRegisterIdentity(
        address[] calldata users,
        IIdentity[] calldata identities,
        uint16[] calldata countries
    ) external;
}

// ONCHAINID identity contract (ERC-734/735)
interface IIdentity {
    // Claims: attestations about the identity from trusted issuers
    function getClaim(bytes32 claimId)
        external view returns (
            uint256 topic,
            uint256 scheme,
            address issuer,
            bytes memory signature,
            bytes memory data,
            string memory uri
        );
    function addClaim(
        uint256 topic,
        uint256 scheme,
        address issuer,
        bytes calldata signature,
        bytes calldata data,
        string calldata uri
    ) external returns (bytes32 claimRequestId);
    function removeClaim(bytes32 claimId) external returns (bool success);
}
```

## Claim Topics Registry & Trusted Issuers

```solidity
// Claim Topics Registry: defines WHICH claim topics are required to hold this token
// e.g. topic 1 = KYC verified | topic 2 = accredited investor | topic 3 = not sanctioned

interface IClaimTopicsRegistry {
    function addClaimTopic(uint256 claimTopic) external;
    function removeClaimTopic(uint256 claimTopic) external;
    function getClaimTopics() external view returns (uint256[] memory);
}

// Standard claim topics (community convention — not standardized):
uint256 constant CLAIM_TOPIC_KYC = 1;
uint256 constant CLAIM_TOPIC_AML = 2;
uint256 constant CLAIM_TOPIC_ACCREDITED_INVESTOR = 3;
uint256 constant CLAIM_TOPIC_PROFESSIONAL_INVESTOR = 4;   // MiFID II
uint256 constant CLAIM_TOPIC_NOT_US_PERSON = 5;           // Reg S
uint256 constant CLAIM_TOPIC_COUNTRY_RESIDENCE = 6;
uint256 constant CLAIM_TOPIC_NOT_SANCTIONED = 7;

// Trusted Issuers Registry: which entities are trusted to issue each claim topic
interface ITrustedIssuersRegistry {
    function addTrustedIssuer(IClaimIssuer trustedIssuer, uint256[] calldata claimTopics) external;
    function removeTrustedIssuer(IClaimIssuer trustedIssuer) external;
    function updateIssuerClaimTopics(IClaimIssuer trustedIssuer, uint256[] calldata claimTopics) external;
    function getTrustedIssuers() external view returns (IClaimIssuer[] memory);
    function isTrustedIssuer(address issuer) external view returns (bool);
    function getTrustedIssuerClaimTopics(IClaimIssuer trustedIssuer)
        external view returns (uint256[] memory);
    function hasClaimTopic(address issuer, uint256 claimTopic) external view returns (bool);
}

// Claim Issuers: KYC providers, AML services, accredited investor verifiers
// They sign claims off-chain and those claims are stored in investor's ONCHAINID
interface IClaimIssuer is IIdentity {
    function isClaimValid(
        IIdentity identity,
        uint256 claimTopic,
        bytes calldata sig,
        bytes calldata data
    ) external view returns (bool);
}
```

## Modular Compliance Engine

```solidity
// IModularCompliance: pluggable compliance rules
// Each module checks one specific rule; all must pass for transfer to succeed

interface IModularCompliance {
    function addModule(address module) external;
    function removeModule(address module) external;
    function isModuleBound(address module) external view returns (bool);
    function getModules() external view returns (address[] memory);

    // Called by token on every transfer
    function canTransfer(address from, address to, uint256 amount) external view returns (bool);
    function transferred(address from, address to, uint256 amount) external;
    function created(address to, uint256 amount) external;
    function destroyed(address from, uint256 amount) external;
}

// Example compliance modules:

// 1. Max Balance per investor
interface IMaxBalanceModule {
    function setMaxBalance(uint256 max) external;
    function getMaxBalance() external view returns (uint256);
}

// 2. Country restriction (whitelist or blacklist)
interface ICountryRestrictModule {
    function addAllowedCountry(uint16 country) external;
    function removeAllowedCountry(uint16 country) external;
    function isCountryAllowed(uint16 country) external view returns (bool);
}

// 3. Max number of investors (e.g. Reg D 2000 limit for 506(b))
interface IMaxInvestorsModule {
    function setMaxInvestors(uint256 max) external;
    function getMaxInvestors() external view returns (uint256);
    function getCurrentInvestors() external view returns (uint256);
}

// 4. Time-lock (Reg D 12-month lock-up)
interface ITimeLockModule {
    function setTimeLock(address investor, uint256 unlockTime) external;
    function getUnlockTime(address investor) external view returns (uint256);
}

// 5. Transfer fees (for secondary market compliance)
interface ITransferFeeModule {
    function setFeeRate(uint256 feeBPS) external;
    function getFeeCollector() external view returns (address);
}
```

## Deployment Pattern

```solidity
// Full T-REX deployment involves deploying:
// 1. IdentityRegistryStorage
// 2. TrustedIssuersRegistry
// 3. ClaimTopicsRegistry
// 4. IdentityRegistry (references 1, 2, 3)
// 5. ModularCompliance + required modules
// 6. Token (references 4 and 5)

// Tokeny provides a TREXFactory for gas-efficient single-transaction deployment:
interface ITREXFactory {
    struct TokenDetails {
        address owner;
        string name;
        string symbol;
        uint8 decimals;
        address irs;                // Identity Registry Storage (can be shared)
        address ONCHAINID;          // Token's own ONCHAINID
        address[] irAgents;         // Identity Registry agents
        address[] tokenAgents;      // Token agents (can mint/burn/freeze)
        address[] complianceModules;
        bytes[] complianceSettings; // ABI-encoded settings per module
    }

    struct ClaimDetails {
        uint256[] claimTopics;
        address[] issuers;
        uint256[][] issuerClaims;   // Which topics each issuer can issue
    }

    function deployTREXSuite(
        string calldata salt,
        TokenDetails calldata tokenDetails,
        ClaimDetails calldata claimDetails
    ) external;
}
```

## ERC-3643 vs ERC-1400 Comparison

| Feature | ERC-3643 (T-REX) | ERC-1400 |
|---|---|---|
| Identity layer | Built-in (ONCHAINID) | External — must integrate separately |
| Compliance | Modular plug-in system | Hook-based, less standardized |
| Partitions | No (single balance) | Yes (multiple tranches) |
| Forced transfer | Yes (built-in) | Yes (controller) |
| Recovery | Yes (built-in) | No standard |
| Batch operations | Yes | No standard |
| Ecosystem | Active (Tokeny, permissioned DeFi) | Older, less active |
| Best for | Single-class equity, bonds | Multi-tranche structured products |

## Integration with DeFi (Permissioned DeFi)

```solidity
// ERC-3643 tokens can integrate with DeFi via compliance-checked wrappers
// Example: only verified investors can provide liquidity

interface IPermissionedPool {
    // Only investors verified in token's identity registry can add liquidity
    function addLiquidity(
        address token,
        uint256 amount,
        address onBehalfOf    // Must be verified investor
    ) external;
}

// Aave Arc, Maple Finance, Centrifuge use similar permissioned DeFi patterns
// Key principle: DeFi smart contract address must itself be whitelisted as
// an eligible holder in the token's identity registry
function whitelistContract(address defiContract, uint16 jurisdictionCode) external {
    // Deploy ONCHAINID for the DeFi contract
    // Issue required claims to its ONCHAINID
    // Register in identity registry
    identityRegistry.registerIdentity(
        defiContract,
        IIdentity(defiContractOnchainID),
        jurisdictionCode
    );
}
```
